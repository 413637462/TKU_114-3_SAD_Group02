### E. 可行性分析

| 可行性類型                        | 分析內容                                                                                                                                                                                                      | 判斷結果                  | 需要調整的範圍                                                                                                                                            |
| --------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **技術可行性**(Technical)   | 核心功能如帳號登入、標籤填寫、任務看板（Todo/Doing/Done）與互評表單，皆可透過常規網頁技術或 Figma 高擬真原型完美展示。然而，若要實作「基於大數據或 AI 的智慧自動配對演算法」，技術門檻與出錯率較高。[cite: 1] | **有風險**[cite: 1] | **縮小範圍**：決定將「機器學習智慧配對演算法」移到範圍外，原型階段改用「簡易條件篩選（如：缺前端就篩選前端）」或「手動分發」來模擬。[cite: 1]       |
| **操作可行性**(Operational) | 學生對於「跨系通識找不到人」以及「組員搭便車」的痛點非常強烈，系統提供的「技能組隊看板」與「搭便車預警」能切中核心需求，使用者使用意願極高，流程比傳統 Google 表單更清晰。[cite: 1]                           | **可行**[cite: 1]   | 無需調整範圍，但需在介面上強化「匿名互評」的隱密性視覺提示，以提升學生信任度。                                                                            |
| **時程可行性**(Schedule)    | 5 週暑修時間極其緊迫。若同時開發完整的課程管理、複雜演算法與即時聊天室，時程絕對會爆炸。專案必須採取「精實原型（MVP）」策略。[cite: 1]                                                                        | **有風險**[cite: 1] | **縮小範圍**：嚴格落實範圍外項目，本期僅聚焦展示「建課規則 -> 標籤組隊 -> 看板分工 -> 互評預警」的核心骨幹流程，其餘次要功能一律不砍不加。[cite: 1] |
| **經濟可行性**(Economic)    | 本專案為暑修課程專案，完全使用免費開源工具（如 Figma 免費版、GitHub、draw.io）進行分析、設計與原型展示，不需要採購伺服器或付費 API。[cite: 1]                                                                 | **可行**[cite: 1]   | 無需調整。團隊投入的成本僅為組員的時間與人力。                                                                                                            |

### F. 風險清單第 1 版

| 風險編號     | 風險描述                                                                                   | 影響                  | 可能性                | 應對方式                                                                                                              |
| ------------ | ------------------------------------------------------------------------------------------ | --------------------- | --------------------- | --------------------------------------------------------------------------------------------------------------------- |
| **R1** | **演算法過於複雜導致範圍蔓延**試圖在 5 週內做出一鍵智慧大數據最佳化分組。            | **高**[cite: 1] | **高**[cite: 1] | 將「真．機器學習智慧配對演算法」移入範圍外，原型改用簡易的條件交叉篩選（如：系所、專長）或直接手動模擬分發。[cite: 1] |
| **R2** | **找不到足夠的真實邊緣人進行測試**導致學生技能檔案的資料樣本不足，難以驗證媒合看板。 | 中                    | 中[cite: 1]           | 在組內與班級進行同儕角色扮演（Role-play），人工捏造 20 筆包含不同系所、專長的模擬學生資料填入系統。[cite: 1]          |
| **R3** | **任務分工看板狀態邏輯失控**若加入過多審批、重做等複雜流程，會導致狀態機過於龐大。   | 中                    | 中[cite: 1]           | 嚴格限制任務看板僅能有三種標準狀態：`Todo`（待辦）、`Doing`（進行中）、`Done`（已完成），不擴充複雜權限。       |
| **R4** | **同儕互評匿名性在原型設計上洩漏**導致學生不敢填寫真實評語，預警系統失效。           | **高**          | 低[cite: 1]           | 在資料物件設計上將互評紀錄（PeerReview）與學生 ID 進行後台解耦，老師端僅能查看圖表與去識別化的文字。                  |
| **R5** | **GitHub 協同作業發生嚴重的程式碼或文件衝突**導致暑修衝刺期檔案毀損或進度倒退。      | 中                    | 中[cite: 1]           | 由主要聯絡人高誠澤制定明確的 GitHub Commit 與 Branch 規範，每次上傳 0707 資料夾前先在本地端進行 Merge 確認。[cite: 1] |

### G. 今天新增的範圍外項目

經過 07/07 的可行性評估與風險分析，為了確保專案不流產，`Group2` 決定正式將以下項目追加至**範圍外（Out-of-Scope）**：

1. **基於大數據或機器學習的智慧自動配對演算法**（本期原型一律改用簡易條件篩選或手動分發模擬）。[cite: 1]
2. **跨學期 / 跨年度的歷史分組數據分析與 Free-rider 黑名單資料庫**（本期系統僅做單一課程內的獨立預警，不進行跨學期的學生歷史表現追蹤）。
