**最後選定題目：**

```
智慧課程分組與跨系組隊媒合系統（Smart Grouping & Cross-Department Team Matching System）
```

**選擇理由：**

1. **避開時程地雷**：學餐系統涉及金流與多端（用戶、店家、後台）即時互動，5 週暑修極易崩潰；而場地登記與教材範例高度重合。
2. **痛點強烈且真實**：組員身為學生，對於「跨系通識課找不到人組隊」以及「專案遇到 Free-rider」有切身之痛，需求訪談與使用者故事極易模擬。
3. **建模邊界清晰**：核心功能聚焦在「多條件規則分組」與「組內任務指派看板」，資料關聯與業務規則明確，非常適合進行 UML 建模與原型展示。

### E. 問題陳述（Problem Statement）

```
對於 大學通識課或大型專題課程的學生與授課老師 而言，目前在 自由組隊與專案執行 的情境中，常發生 「邊緣人找不到組別、跨系合作資訊不對等、以及 Free-rider（搭便車）隱匿其中」 的痛點與影響。

本專案希望設計一個 智慧課程分組與跨系組隊媒合系統，協助 老師一鍵依規則自動分組、學生透過技能標籤精準尋找隊友，並讓組內任務分工完全透明化，以在本期聚焦於 「多條件自動分組、學生技能媒合看板、以及組內任務指派與預警流程」，不處理校務正式成績系統的雙向串接。
```

### F. 專案目標（Project Goal）

在 5 週內建立一套整合「智慧規則組隊」與「專案任務追蹤」的網頁系統原型。旨在降低老師處理落單學生的行政時間，並透過透明化的看板與同儕互評機制，提升跨系組隊的媒合成功率，同時降低組內黑箱分工與搭便車的風險。

---

### G. 範圍內項目（In-Scope）

1. **使用者與技能檔案管理**：學生可編輯個人技能標籤（前端、後端、簡報、報告撰寫）與期望隊友類型。
2. **課程與分組規則設定**：老師可建立課程，並自訂分組條件（每組人數上下限、是否強制跨系、截止時間）。
3. **智慧組隊與媒合看板**：提供「組長招募隊員」與「個人尋找隊伍']的雙向媒合看板，並提供一鍵自動配對落單學生的功能。
4. **組內任務分工看板**：組長可建立任務（Todo / Doing / Done），指派負責人並設定截止日期，成員可回報進度。
5. **匿名同儕互評與預警系統**：期中/期末進度檢查時，組員可進行匿名互評，系統針對進度嚴重落後或評價過低的成員發出「搭便車預警」提示給老師與組長。

---

### H. 範圍外項目（Out-of-Scope）

1. **校務正式選課系統串接**：不與學校的教務系統進行 API 雙向同步，課程與學生名單採 Excel/CSV 匯入模擬。
2. **學校正式單一登入（SSO）**：不串接學校正式帳號驗證，改用自建的模擬帳號密碼登入。
3. **即時通訊聊天室**：系統內不開發原生 Chat 功能，僅提供填寫 Discord / Line 群組外部連結的欄位。
4. **雲端檔案儲存空間**：不提供專案報告的檔案上傳與空間管理，改以提供 Google Drive 共同編輯連結取代。
5. **正式成績與學分系統整合**：互評數據僅在系統內產出報表供老師參考，不直接寫入學校的正式期末成績資料庫。

### K. 初步成功標準（Initial Success Criteria）

1. **架構可追蹤性**：期末的原型（Prototype）畫面中，每一顆按鈕與功能（如：一鍵媒合、搭便車預警）都能精準回溯到 SRS 需求文件中的特定使用者故事與驗收條件。
2. **完整端到端流程（End-to-End）**：在期末展示時，能流暢示範「老師建課 -> 學生靠標籤智慧組隊 -> 組長分派任務看板 -> 匿名互評產出預警報表」的完整閉環流程，中途無瀏覽器報錯。
3. **範圍控制標準**：嚴格杜絕範圍蔓延，團隊能清楚向評審教授說明哪些功能（如即時通訊、串接校務系統）屬於 Out-of-Scope，並用最聰明的替代方案（如外部連結）在 5 週內優雅解決。
