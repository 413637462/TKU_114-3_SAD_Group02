### C. 專案章程第 2 版（Project Charter v2）

#### 1. 專案題目（Project Title）

智慧課程分組與跨系組隊媒合系統（Smart Grouping & Cross-Department Team Matching System）

#### 2. 目標使用者（Target Users）

| 角色                        | 主要任務                                                                               | 主要痛點                                                                                                                        |
| --------------------------- | -------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| **授課老師 / 助教**   | 發布課程、設定跨系分組條件與人數限制、追蹤各組分工進度、下載期末同儕互評去識別化報表。 | 自由組隊導致大量落單邊緣人需人工安置，耗時耗力；傳統紙本或 Google 表單難以掌握小組內部的真實分工狀態。                          |
| **一般學生 / 邊緣人** | 填寫個人技術與軟實力標籤、在系統看板中尋找志同道合或技能互補的隊伍。                   | 跨系通識大班不認識旁人，分組時面臨極大的人際尷尬；資訊不對等，容易誤入缺乏核心技能的雷組。                                      |
| **專案組長**          | 建立隊伍並發布殘缺技能招募、建立組內任務分工看板、指派待辦事項給成員。                 | 用 Line 溝通易被閒聊洗版，難以明確追蹤成員進度；期末同儕互評流於形式，缺乏客觀數據制衡不付出的搭便車者（Free-rider）[cite: 1]。 |

#### 3. 問題陳述（Problem Statement）

對於 大學通識課或大型跨系專題課程的學生與授課老師 而言，目前在 自由組隊與專案執行 的情境中，常發生 「邊緣人找不到組別、跨系合作資訊不對等、以及 Free-rider（搭便車）隱匿其中」 的痛點與影響[cite: 1]。

本專案希望設計一個 智慧課程分組與跨系組隊媒合系統，協助 老師一鍵依規則自動分組、學生透過技能標籤精準尋找隊友，並讓組內任務分工完全透明化，以在本期聚焦於 「多條件自動分組、學生技能媒合看板、以及組內任務指派與預警流程」，不處理校務正式成績系統的雙向 API 串接[cite: 1]。

#### 4. 專案目標（Project Goal）

在 5 週暑修時程內，建立一套整合「多條件規則組隊」與「透明化任務追蹤」的精實網頁系統原型（MVP）[cite: 1]。旨在降低管理端安置落單學生的行政成本，並透過技能檔案看板提升跨系媒合率，同時藉由看板分工與匿名同儕互評機制，徹底降低組內搭便車的黑箱風險[cite: 1]。

#### 5. 利害關係人摘要（Stakeholder Summary）

本專案利害關係人包含直接操作系統的授課老師、一般學生、專案組長與課程助教[cite: 1]。同時納入不直接操作系統、但具備合規與資安否決權的間接利害關係人：學校資訊處（關心資料合規與模擬帳號資安）與教務處/系辦承辦人（負責官方選課名單與最終成績結算限制）[cite: 1]。系統將採取權力/關注矩陣之分層策略進行需求收斂[cite: 1]。

#### 6. 範圍內項目（In-Scope）

1. **使用者與技能檔案管理**：學生可編輯個人技能標籤（如前端、簡報、企劃）與期望隊友類型[cite: 1]。
2. **課程與分組規則設定**：老師可自訂分組限制條件（每組人數上下限、是否強制跨系所、截止時間）[cite: 1]。
3. **智慧組隊與媒合看板**：提供「組長招募殘缺技能」與「個人尋找隊伍」的雙向看板，並提供簡易條件配對功能[cite: 1]。
4. **組內任務分工看板**：組長可建立任務物件，指派負責人並設定截止日期，成員可進行看板狀態變更（Todo/Doing/Done）[cite: 1]。
5. **匿名同儕互評與預警系統**：期中/期末進度檢查時，組員可進行匿名互評，系統針對異常數據自動發出「搭便車預警」提示給老師與組長[cite: 1]。

#### 7. 範圍外項目（Out-of-Scope）

1. **校務正式選課系統 API 雙向串接**：名單採模擬 Excel/CSV 檔案單向匯入[cite: 1]。
2. **學校正式單一登入（SSO）帳號驗證**：改用系統內部自建模擬帳號[cite: 1]。
3. **原生即時通訊聊天室與討論區**：不投入開發資源，僅提供填寫外部 Discord / Line 群組連結之文字欄位[cite: 1]。
4. **雲端檔案儲存空間管理**：不提供報告檔案上傳，改以提供 Google Drive 共同編輯外部連結取代[cite: 1]。
5. **基於大數據或機器學習的真實智慧自動配對核心演算法**（因應 07/07 可行性評估移除）[cite: 1]。
6. **跨學期 / 跨年度的歷史信用黑名單大數據資料庫**[cite: 1]。

#### 8. 原型核心任務（Core Tasks for Prototype）

| 任務                                | 主要角色        | 任務起點                                                            | 任務終點                                                               | 初步驗收標準                                                                                        |
| ----------------------------------- | --------------- | ------------------------------------------------------------------- | ---------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------- |
| **1. 建立課程與分組規則**     | 授課老師        | 老師登入系統，點選「新增課程」。                                    | 成功發布課程並產生「邀請碼」與「限制條件物件」。                       | 系統能正確攔截並不允許不符條件（如未跨系）的學生強行手動組隊[cite: 1]。                             |
| **2. 技能標籤填寫與組隊媒合** | 一般學生        | 邊緣人學生登入填寫專長標籤，並輸入邀請碼進入課程。                  | 學生點選「簡易條件配對」，系統依缺額與技能將其手動或半自動分配至隊伍。 | 學生成功加入隊伍，且該隊伍的人數與系所組成完全符合老師設定的門檻規則[cite: 1]。                     |
| **3. 任務看板指派與分工追蹤** | 專案組長 / 組員 | 組長在隊伍後台點選「建立新任務」，輸入內容並指派給組員 A[cite: 1]。 | 組員 A 點選進度更新，將任務拖曳至「已完成」，組長與老師後台同步更新。  | 看板能即時反映各成員名下的任務總數與完成率，並產出分工比例圖，供同儕匿名互評預警機制讀取[cite: 1]。 |

#### 9. 初步成功標準（Initial Success Criteria）

1. **架構可追蹤性（Traceability）**：期末的原型展示畫面中，每一顆核心按鈕與邏輯功能（如：條件配對、搭便車預警、看板流轉）都能精準回溯到需求文件（SRS）中的特定使用者故事與驗收條件[cite: 1]。
2. **完整端到端流程（End-to-End）**：在期末 Demo 時，能流暢示範「老師建課規則 -> 學生靠標籤組隊 -> 組長分派任務看板 -> 匿名互評產出預警報表」的完整閉環，中途無瀏覽器報錯[cite: 1]。
3. **範疇控制率**：嚴格控制範圍蔓延，團隊能清楚向評審教授說明哪些功能屬於 Out-of-Scope，並用最低成本的替代方案在 5 週內優雅解決，確保原型交付品質[cite: 1]。

#### 10. 主要風險與應對方式（Major Risks and Responses）

| 風險                                                       | 影響         | 可能性       | 應對方式                                                                                                                                   |
| ---------------------------------------------------------- | ------------ | ------------ | ------------------------------------------------------------------------------------------------------------------------------------------ |
| **智慧演算法過於複雜導致範圍蔓延（Scope Creep）**    | **高** | **高** | 將「真．機器學習智慧配對演算法」移入範圍外，原型階段改用簡易的關鍵字與系所欄位交叉篩選，以介面邏輯模擬完成[cite: 1]。                      |
| **任務分工看板狀態機邏輯失控（State Machine Risk）** | 中           | 中           | 嚴格限制原型中的任務看板僅能有三種標準狀態：`Todo`（待辦）、`Doing`（進行中）、`Done`（已完成），不擴充複雜的行政審核權限[cite: 1]。 |
| **資料架構設計不當導致匿名互評外洩**                 | **高** | 低           | 在資料物件與 ERD 設計上，確保互評紀錄（PeerReview）採去識別化設計，老師端僅能查看圖表與平均分，組員後台完全無法回溯評分者 ID[cite: 1]。    |

### D. 三分鐘短講提案稿

```text
我們小組的題目是：智慧課程分組與跨系組隊媒合系統。

目標使用者是：大學通識課、跨系大班專題課程的授課老師、一般學生與專案組長。

目前的問題是：自由組隊常導致落單的邊緣人極度尷尬、跨系合作時技能資訊不對等、以及缺乏客觀數據防範黑箱分工與搭便車（Free-rider）現象。

本系統希望改善：透過技能標籤透明化提升媒合成功率，並利用任務看板與匿名互評機制讓團隊分工完全公開透明。

本期範圍內包含：使用者技能檔案管理、分組限制規則設定、雙向組隊媒合看板、組內任務分工看板（Todo/Doing/Done）以及匿名同儕互評預警系統。

本期範圍外包含：校務選課系統雙向 API 串接、學校正式單一登入（SSO）、原生即時通訊聊天室、以及真實的機器學習自動配對核心演算法。

期末原型會展示三個任務：
1. 授課老師登入系統建立課程並設定跨系人數限制規則。
2. 學生填寫個人技能檔案，透過限制條件配對順利加入互補隊伍。
3. 組長建立專案任務看板並指派給組員，組員變更狀態後，後台同步更新分工比例並提供同儕互評預警。

目前最大的風險是：智慧自動配對演算法與看板狀態邏輯若設計得太複雜，會導致 5 週衝刺期範圍蔓延、專案無法按時交付。

我們的應對方式是：嚴格執行範疇控制，正式剔除機器學習演算法與原生聊天功能，看板狀態機嚴格限制在三種常規狀態內，確保期末原型端到端流暢展示。
```
